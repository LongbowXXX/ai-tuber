# 制約・落とし穴・非機能要件

## 1. 非機能要件と考慮事項

- **リアルタイム性**: AI (`vtuber-behavior-engine`) から `vtube-stage` へのツール呼び出し、および `vtube-stage` での VRM モデル更新は、スムーズなアニメーションを実現するために低遅延である必要があります。HTTP/SSE は十分な応答性を持ちますが、AI の応答生成 (`vtuber-behavior-engine`) は最も遅延が大きいボトルネックとなる可能性があり、最適化が必要です。
- **パフォーマンス**: `vtube-stage` での Three.js レンダリングは、特に複雑なモデルやエフェクトを使用する場合、効率的である必要があります。GPU アクセラレーションを活用し、不要な再レンダリングを避けることが重要です。
- **拡張性**:
  - AI 能力: ADK/MCP の採用により、新しいツールやエージェントを `vtuber-behavior-engine` に追加して機能を拡張することが容易になります。
  - キャラクター: 新しい VRM モデルを追加し、関連する設定（`vtube-stage` での感情マッピングなど）を行うことで、複数のキャラクターをサポートできます。
- **保守性**: コンポーネント間の明確な分離と、ADK による構造化された AI ロジックの実装は、システムの保守性を向上させます。各コンポーネントは独立してテストおよびデプロイできます。
- **エラーハンドリング**: 各コンポーネントは、通信エラー（MCP 接続失敗、OBS 接続失敗）や内部エラー（AI モデルエラー、レンダリングエラー）を適切に処理し、可能な限り回復するか、明確なエラーログを出力する必要があります。

## 2. デプロイメントアーキテクチャ (概要)

- **AI Backend**: `vtuber-behavior-engine` は Python アプリケーションであり、ローカルマシンまたはコンテナ環境で実行できます。
- **Desktop App**: `vtube-stage` は Electron デスクトップアプリケーションであり、ローカルマシン上で実行されます。
- **通信**:
  - `vtuber-behavior-engine` と `vtube-stage` は HTTP/SSE (`http://localhost:8080/sse`) で MCP 通信します。
  - `vtube-stage` と OBS Studio はウィンドウキャプチャで連携します（将来的に obs-websocket 対応）。

## 3. 既知の制約と落とし穴

- **認証/認可がない**: `vtube-stage` の MCP Server (`http://localhost:8080/sse`) は、現状ローカル用途を想定したシンプル構成で、アクセス制御がありません。
- **同期/順序の制約**: `speak` ツールは音声再生完了まで待機します。`vtube-stage` 側で処理が失敗すると、AI 側がタイムアウトする可能性があります。
- **OBS 連携**: obs-websocket による OBS 制御は現状未実装です（ウィンドウキャプチャ前提）。
- **データ検証**: 新しい MCP ツールを追加した場合、AI 側とステージ側の両方で型定義とバリデーションを更新する必要があります。
