## 📌 バージョニング関連

### セマンティックバージョニング (Semantic Versioning / SemVer)

ソフトウェアのバージョン番号を `MAJOR.MINOR.PATCH` の形式で管理する規則。

**形式**: `X.Y.Z` (例: `1.2.3`)

- **MAJOR (X)**: 破壊的変更（後方互換性なし）
  - 例: API の仕様変更、既存機能の削除
- **MINOR (Y)**: 機能追加（後方互換性あり）
  - 例: 新機能の追加、既存機能の拡張
- **PATCH (Z)**: バグ修正（後方互換性あり）
  - 例: バグ修正、セキュリティパッチ

**参考**: [semver.org](https://semver.org/lang/ja/)

## 📌 プロジェクト管理関連

### ストーリー

機能追加・変更またはリリース全体を表す親チケット（Epic）。複数のタスクで構成される。

**種類**:

- **機能追加・変更ストーリー**: 新機能の開発や既存機能の変更
- **リリースストーリー**: バージョンリリース作業全体

### タスク

ストーリーを構成する個別の作業単位。サブチケット（子チケット）として管理される。

### ExitCriteria (完了基準)

ストーリーやタスクが完了したと判断するための具体的な条件。すべての項目が満たされることで完了と見なされる。

**例**:

- リリース前確認がすべて実施され、パスしている
- リリースページが作成されている
- すべてのドキュメントが最新の状態に更新されている

### 依存関係

タスク間の順序関係。あるタスクが完了しないと次のタスクを開始できない関係を示す。

**表記**:

- **前提**: このタスクを開始する前に完了している必要があるタスク
- **後続**: このタスクが完了した後に開始できるタスク

## 📌 品質管理関連

### サニティテスト (Sanity Test)

リリース前に実施する基本的な動作確認テスト。主要機能が正常に動作することを確認する。

**目的**:

- クリティカルなバグの早期発見
- リリース可否の判断材料
- 品質の最低ラインの保証

**類似用語**: スモークテスト (Smoke Test)

### 脆弱性チェック

依存ライブラリやコードに既知のセキュリティ脆弱性が含まれていないか確認する作業。

**ツール例**:

- Python: `pip-audit`, `safety`
- Node.js: `npm audit`, `yarn audit`

### 静的解析 (Static Analysis)

ソースコードを実行せずに解析し、バグやコード品質の問題を検出する手法。

**検出内容**:

- コーディング規約違反
- 潜在的なバグ
- セキュリティ上の問題
- コードの複雑度

**ツール例**:

- 汎用: Coverity, SonarQube, CodeQL
- Python: Pylint, Flake8, mypy
- JavaScript/TypeScript: ESLint, TypeScript Compiler

## 📌 成果物管理関連

### Git 管理

バージョン管理システムでファイルを管理すること。変更履歴が追跡され、レビュープロセスを経て更新される。

**対象成果物**:

- ソースコード
- 仕様書、設計書
- ドキュメント (README, CHANGELOG 等)
- テスト項目
- ライセンス一覧

### プロジェクト管理ツール管理

JIRA、Redmine、GitHub Issues 等のツール上で記録・管理すること。主に確認結果や進捗状況を記録する。

**対象成果物**:

- チェック結果（脆弱性、静的解析等）
- テスト実施結果
- ExitCriteria 文書
- 確認記録

## 📌 リリース関連

### リリースノート

新バージョンの変更内容をユーザーに伝えるドキュメント。

**構成要素**:

- **Added**: 新機能
- **Changed**: 変更内容
- **Deprecated**: 非推奨になった機能
- **Removed**: 削除された機能
- **Fixed**: バグ修正
- **Security**: セキュリティ修正

**参考**: [Keep a Changelog](https://keepachangelog.com/ja/)

### アーティファクト (Artifact)

リリースに含まれる成果物。実行可能ファイル、パッケージ、ドキュメント等。

**例**:

- バイナリファイル (`.exe`, `.app`)
- パッケージ (`.whl`, `.tar.gz`)
- ソースコードアーカイブ (`.zip`)

### パッケージレジストリ

ソフトウェアパッケージを公開・配布するためのリポジトリ。

**例**:

- Python: PyPI (Python Package Index)
- Node.js: npm (Node Package Manager)
- Ruby: RubyGems

## 📌 ライセンス関連

### コピーレフト

派生作品も同じライセンスで公開することを要求するライセンス条項。

**代表的なライセンス**:

- GPL (GNU General Public License)
- AGPL (Affero GPL)
- LGPL (Lesser GPL) - 部分的コピーレフト

### パーミッシブライセンス

制約が少なく、商用利用や再配布が自由に行えるライセンス。

**代表的なライセンス**:

- MIT License
- Apache License 2.0
- BSD License

## 📌 その他

### WBS (Work Breakdown Structure)

作業を階層的に分解した構造図。プロジェクト全体の作業を視覚化する。

### マイルストーン

プロジェクトの重要な中間目標地点。リリースやフェーズ完了等。

### コンフリクト (Conflict)

複数の変更が競合し、自動的にマージできない状態。手動での解決が必要。
