## 概要

本ドキュメントでは、プロジェクトの各タスクで作成される成果物を一覧化しています。

成果物の管理方針の詳細は[#成果物の管理原則](#成果物の管理原則)を参照してください。

## 成果物の管理原則

### 1. Git 管理

ソースコードやドキュメントなど、プロジェクトの技術的資産は Git で管理され、変更履歴が追跡可能である必要があります。

**Git 管理対象**:

- ソースコード
- テストコード
- 仕様書、設計書
- サニティテスト項目(大きなファイルは Git LFS を利用)
- README、CHANGELOG、API 仕様書などのドキュメント
- ライセンス一覧ファイル

**プロジェクト管理ツール管理対象** (Git 管理不要):

- バージョン名の合意記録
- 関連修正の反映確認記録
- ExitCriteria 文書
- 機能実装完了確認項目
- テスト実施結果(サニティテスト、機能確認、リリース前確認)
- 品質チェック結果(脆弱性、静的解析)
- リリース確認項目と実施結果
- ExitCriteria 確認結果

### 2. 命名規則

- **ファイル名**: スネークケース（例: `feature_checklist_auth.md`）
- **ディレクトリ名**: ケバブケース（例: `exit-criteria`）
- **バージョン番号**: セマンティックバージョニング（例: `v1.2.3`）

### 3. フォーマット

- **ドキュメント**: Markdown 形式を推奨
- **設計図**: PlantUML, Mermaid 等のテキストベース
- **データ**: CSV, JSON 形式を推奨（可読性と diff 性）
- **サニティテスト項目**: Excel (`.xlsx`), Markdown (`.md`), CSV (`.csv`), Google Sheets 等、プロジェクトに応じて選択
  - 大きなファイルの場合は Git LFS を利用
  - 常に最新のアプリ仕様に合わせた単一ファイルで管理

### 4. レビュー

Git で管理される成果物は以下のレビュープロセスを経る必要があります:

1. プルリクエストの作成
2. チームメンバーによるレビュー
3. 承認後のマージ

## ディレクトリ構造例

```
ai-tuber-system/
├── docs/
│   ├── specs/                      # 仕様書
│   │   └── feature_xxx.md
│   ├── design/                     # 設計書
│   │   └── architecture.md
│   ├── test/                       # テスト関連
│   │   └── sanity_checklist.*      # サニティテスト項目
│   ├── licenses.md                 # ライセンス一覧
│   └── workflow/                   # 作業フロー(本ドキュメント)
├── packages/
│   ├── vtuber-behavior-engine/
│   │   ├── src/                    # ソースコード
│   │   └── tests/                  # テストコード
│   ├── stage-director/
│   │   ├── src/
│   │   └── tests/
│   └── vtube-stage/
│       ├── src/
│       └── tests/
├── README.md                       # プロジェクト概要
├── CHANGELOG.md                    # 変更履歴
└── LICENSE
```

**注意**: テスト実施結果、ExitCriteria 文書、品質チェック結果はプロジェクト管理ツール上で管理するため、Git リポジトリには含まれません。
